Portfolio Site

The opportunity to learn about front end web development has allowed better understanding of the languages making up products I design. Additionally, the chance to create my own portfolio website gave much needed insight into code limitations, capabilities, and errors to be considered while designing in collaboration with a front end development team. Moreover this project has served as an opportunity to refine what I want from my portfolio site and the impression I want to give to anyone interested in my work.

Problem | Context | Goals

Problem

As a UX professional I need a way to showcase and explain my projects in an engaging and narrative way in order to market myself and my skills. Having an easily accessible, well designed and professional website allows anyone to view my work with explanation into my design decisions and creative concept.

Context

As a supplementary part to my UX design course, the front end development for designers allows better understanding of HTML, CSS, and JS in the context of front end development, design limitation and capabilities from a UX and dev perspective.

Goals

  • Live, fully functional website
  • Well formatted, easily readable code
  • Responsive design

    • Research

    When searching for UX portfolios that I liked and wanted to aspire to, I found that I leaned towards visually striking portfolios with visual personality and compelling story. Rather than relying on maximal information, I also found that I enjoyed more minimalistic portfolios with emphasis on quality of information rather than quantity.

    During the creation of my initial portfolio and while creating my current portfolio I realized that I was providing an overwhelming amount of information but lacking quality information. In my second attempt I cut down on the information supplied and focused on design decisions.

    What makes a good portfolio?

    Design

    Based on my UX project; Ripple I wanted a design that was fairly minimalist with vibrant colors. I chose a contrasting color scheme and incorporated a more artsy feel to my logo that reflects more of my personality.

    HTML Process

    Starting with a blank slate in VScode I followed provided instructions to create my index.html page using HTML5.

    Through the CF instructions I was able to slowly build the basic HTML infrastructure for the work section, footer, and about page. Once I started getting more comfortable with HTML I added a resume option in the navigation bar based on user feedback.

    During this stage I struggled with understanding how HTML was actually formatted-- initially I didn’t understand the indentation style but this was resolved by talking about it with my mentor,

    Snapshot of HTML code for my portfolio website

    CSS Process

    For styling colors, text, columns and rows I utilized an external CSS document as opposed to inline CSS for readability

    With CSS in specific there was a lot of trial and error in getting buttons to react in the way I wanted. Additionally, I struggled with media queries. With help I figured out how to optimize for display size and writing rules that would effect specific parts of the website.

    JS Process

    In all honesty I had had little to no idea what JS did for websites. Understanding how JS adds fundamental website behavior and more complex features is crucial when designing and coding responsive websites. Especially now when most people use two or more devices of different sizes.

    In my portfolio site JS was used exclusively to add responsive design to the navigation menu for better user experience while on a mobile device.

    Responsive design in action

    Usability Testing

    Usability testing was completed with 5 people in the age range of 21-32 who are comfortable with technology.

    I asked each person to complete three different tasks to determine how clear the navigation and visual hierarchy is on my portfolio site. I also asked 5 personal questions to determine their relationship and comfort level with technology. The tasks were followed by four questions pertaining to the difficulty and mental chatter they experienced.

    Overall testers were able to navigate the site very easily. There were some issues and questions surrounding where the resume is on the website so I added a link to a PDF version in different spots. Additionally, I changed “work” to “home”.

    Tester demographics
    Usability testing results

    Cross Browser Testing

    Using BrowserStack I checked to see if my website would work on different browsers and devices. The only issue I had was the Ripple case study menu not being responsive. This turned out to be a simple code issue. Otherwise nothing came up for other devices and browsers.

    Code Quality and Linters

    In order to check and double check my code for quality, readability, and organization I opted to install four different code linters. Although Stylelint and StandardJS threw errors I didn’t really understand I was able to resolve the errors via a different Javascript validator (JS Validator) and the CSS validator from W3C. Otherwise I checked my HTML via the W3C markup validation.

    I also checked accessibility with my color scheme and got a minimum rating of 8.69 and a maximum of 15.81, both of which are AA quality.

    Launch

    After many checks and rechecks I finally felt ready to launch my website. My goal of creating a easily accessible, well designed and professional website is almost fully realized. There are still some changes I want to push content and design wise but for the moment (and the sake of time) I’m very pleased overall with the quality and content I’ve managed to produce. The front end development course was very challenging for me but I’m still looking forward to practicing website development alongside UX.
